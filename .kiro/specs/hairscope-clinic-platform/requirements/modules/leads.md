# Leads Management

> Covers: Lead capture (manual, webhook, selfie analysis), lead assignment, CRM actions, status pipeline, and conversion to Patient.
> Events emitted: `LeadCreated`, `LeadConverted`
> Events consumed: none (lead creation is a source event)

---

## Glossary

- **Lead**: A prospective patient who has not yet been converted to a Patient record.
- **Lead_Source**: `MANUAL` | `WEBHOOK` | `SELFIE_ANALYSIS` — see `shared/enums.md`.
- **Lead_Status**: `NEW` | `CONTACTED` | `QUALIFIED` | `CONVERTED` | `LOST` — see `shared/enums.md`.
- **Lead_Priority**: `LOW` | `MEDIUM` | `HIGH` | `URGENT` — see `shared/enums.md`.
- **Lead_Tag**: A free-text label attached to a Lead for categorization and filtering.
- **Lead_Action**: A logged interaction with a Lead including action type, optional status change, and content text.
- **Webhook_Source**: An external campaign system configured to push Lead data to the platform via HTTP POST. Configured by Organization_Admin only.
- **Field_Mapping**: A per-Webhook_Source configuration that maps incoming webhook payload fields to Lead profile fields.
- **Selfie_Analysis_Report**: The AI hair-loss assessment PDF generated by the Selfie Analysis web component.
- **Unassigned_Lead**: A Lead with no `clinicId` assigned. Can originate from Selfie Analysis (visitor did not select a clinic), webhooks (no clinic in payload), or manual creation by an Organization_Admin without selecting a clinic. Visible to Organization_Admin only until a clinic is assigned.
- **Round_Robin_Assignment**: The mechanism by which incoming leads within a Clinic are distributed evenly across active Staff members with lead access, in rotation order.
- **Conversion**: The process of creating a Patient record from a Lead record, transferring applicable fields automatically.

---

## Lead Assignment Model

```
Lead created
    │
    ├── Source: MANUAL by Clinic Staff → assigned to current Clinic, distributed via Round_Robin
    ├── Source: MANUAL by Organization_Admin → Admin selects Clinic (optional); if no Clinic selected → Unassigned_Lead
    ├── Source: SELFIE_ANALYSIS → visitor selects Clinic from list; if no Clinic selected → Unassigned_Lead
    └── Source: WEBHOOK → Clinic determined by field mapping; if no Clinic in payload → Unassigned_Lead

Unassigned_Lead
    └── Organization_Admin assigns Clinic → Lead becomes clinic-scoped → Round_Robin distributes to Staff

Within Clinic
    └── Round_Robin assigns Lead to Staff member
    └── Clinic_Admin can reassign Lead to another Staff if status is NEW or LOST
```

## Requirements

### LM-1: Lead Profile

**User Story:** As a Staff member with lead create permission, I want to create and manage lead profiles so that the clinic can track all prospective patients in one place.

#### Acceptance Criteria

1. THE Platform SHALL store the following fields for each Lead: `name`, `age`, `gender`, `email`, `phone`, `createdAt`, `status`, `priority`, `tags[]`, `leadSource`, `sourceDetail` (page URL or campaign ID), `clinicId` (nullable for unassigned leads), `assignedStaffId` (nullable).
2. THE Platform SHALL allow duplicate Leads — the same email or phone MAY exist across multiple Lead records.
3. THE Platform SHALL allow Staff to update `status`, `priority`, and `tags` at any time.
4. WHEN a Lead is created via the Selfie_Analysis web component, THE Platform SHALL attach the selfie analysis questions, answers, and `Selfie_Analysis_Report` to the Lead profile.
5. WHEN a Lead is created, THE Platform SHALL set the initial `status` to `NEW`.
6. WHEN a Lead is created, THE Platform SHALL emit a `LeadCreated` event.
7. WHEN a Lead is created, THE Platform SHALL record the action in the Audit_Log.
8. Clinic-level Staff SHALL only see Leads belonging to their Clinic (`clinicId` matches their Clinic).
9. Organization_Admins SHALL see all Leads across all Clinics within their Organization, including Unassigned_Leads.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Missing `name` field | `VALIDATION_ERROR` (field: `name`) |

#### Correctness Properties

- Two Lead records with identical email values SHALL both be creatable and retrievable without error.
- For any newly created Lead L: `L.status = NEW` immediately after creation.
- For any Clinic-level Staff member S in Clinic C: the Lead list query SHALL only return Leads where `clinicId = C`.
- For any Organization_Admin A: the Lead list query SHALL return all Leads in A's Organization including those with `clinicId = null`.

### LM-2: Manual Lead Entry

**User Story:** As a Staff member with lead create permission, I want to manually enter lead information so that I can capture prospective patients from phone calls, walk-ins, and other offline channels.

#### Acceptance Criteria

1. THE Platform SHALL provide a `createLead` mutation for Staff to manually create a Lead.
2. `name` is the only required field; all other fields are optional.
3. WHEN a Clinic-level Staff member creates a Lead manually, THE Platform SHALL automatically set `clinicId` to the Staff member's current Clinic and distribute the Lead via Round_Robin to an active Staff member with lead access.
4. WHEN an Organization_Admin creates a Lead manually, THE Platform SHALL present a Clinic selector. If a Clinic is selected, `clinicId` is set and Round_Robin distributes the Lead. If no Clinic is selected, the Lead is created as an Unassigned_Lead.
5. WHEN a Lead is manually created, THE Platform SHALL set `leadSource` to `MANUAL`.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Missing `name` | `VALIDATION_ERROR` (field: `name`) |

---

### LM-3: Webhook Lead Ingestion

**User Story:** As an Organization_Admin, I want to configure webhooks from external marketing campaigns so that leads are automatically captured in the platform.

#### Acceptance Criteria

1. THE Platform SHALL allow Organization_Admins (only) to configure one or more Webhook_Sources, each with a unique endpoint URL and a Field_Mapping.
2. THE Field_Mapping SHALL allow mapping each incoming webhook payload field to the corresponding Lead profile field, including an optional `clinicId` mapping.
3. WHEN a webhook payload is received at a configured endpoint, THE Platform SHALL parse it using the configured Field_Mapping and create a Lead record.
4. WHEN a webhook payload is received, THE Platform SHALL set `leadSource` to `WEBHOOK` and `sourceDetail` to the campaign ID from the mapped field.
5. IF the Field_Mapping includes a `clinicId` and the payload contains a valid Clinic ID, THE Platform SHALL assign the Lead to that Clinic and distribute via Round_Robin.
6. IF the Field_Mapping does not include a `clinicId` or the payload does not contain one, THE Platform SHALL create the Lead as an Unassigned_Lead.
7. IF a webhook payload is missing a required mapped field, THE Platform SHALL reject the payload, return an error response to the sender, and record the failure in the Audit_Log.
8. Webhook configuration (create, edit, delete Webhook_Sources) is restricted to Organization_Admins only.

#### Failure Cases

| Condition | HTTP Status | Error Code |
|-----------|-------------|------------|
| Invalid API key | 401 | `WEBHOOK_INVALID_API_KEY` |
| Missing required mapped field | 422 | `WEBHOOK_MAPPING_ERROR` |
| Malformed JSON payload | 400 | `WEBHOOK_PAYLOAD_INVALID` |
| Webhook source not found | 404 | `NOT_FOUND` |
| Clinic_Admin attempting to configure webhooks | 403 | `FORBIDDEN` |

#### Correctness Properties

- For any webhook payload P with field mapping M, the Lead created from P SHALL have field values equal to `M(P)` for every mapped field.
- For any webhook payload P missing a field required by mapping M, the Lead creation SHALL fail and no Lead record SHALL be created.
- Webhook_Source configuration mutations SHALL fail for any actor who is not an Organization_Admin.

### LM-4: Selfie Analysis Web Component Lead Capture

**User Story:** As a prospective patient on a clinic's website or app, I want to complete a selfie-based hair analysis and select my preferred clinic so that I receive an instant assessment and the right clinic can follow up with me.

#### Acceptance Criteria

1. THE Platform SHALL provide a Selfie_Analysis web component built with Stencil, embeddable on any Organization's website or app.
2. THE Selfie_Analysis web component is authenticated using the Organization's API key. There are no clinic-level API keys.
3. WHEN a visitor starts the Selfie_Analysis flow, THE Platform SHALL display a list of Clinics belonging to the Organization, allowing the visitor to select their preferred Clinic (e.g., by location proximity or convenience).
4. WHEN a visitor completes the Selfie_Analysis flow, THE Platform SHALL capture: `name`, `age`, `gender`, `email`, `phone`.
5. WHEN a visitor completes the Selfie_Analysis flow and has selected a Clinic, THE Platform SHALL generate a `Selfie_Analysis_Report`, create a Lead record assigned to the selected Clinic, and distribute the Lead via Round_Robin to an active Staff member with lead access in that Clinic.
6. WHEN a visitor completes the Selfie_Analysis flow without selecting a Clinic, THE Platform SHALL create the Lead as an Unassigned_Lead visible only to the Organization_Admin.
7. THE Platform SHALL attach the selfie analysis questions, answers, and `Selfie_Analysis_Report` to the created Lead record.
8. WHEN the `Selfie_Analysis_Report` is generated, THE Platform SHALL offer the visitor the option to book an appointment via the Appointment Booking web component.
9. Setup and configuration of the Selfie_Analysis web component and its API key are restricted to Organization_Admins only.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Invalid or missing Organization API key | `WEBHOOK_INVALID_API_KEY` |
| Missing required visitor fields | `VALIDATION_ERROR` |

#### Correctness Properties

- For every completed Selfie_Analysis flow with a Clinic selected, exactly one Lead record SHALL be created with `clinicId` set to the selected Clinic.
- For every completed Selfie_Analysis flow without a Clinic selected, exactly one Unassigned_Lead SHALL be created with `clinicId = null`.
- For any Lead L created via Selfie_Analysis: `L.selfieAnalysisReport` is non-null.
- The Clinic list displayed to the visitor SHALL only include active Clinics belonging to the Organization.

### LM-5: Lead Assignment and Round-Robin Distribution

**User Story:** As a Clinic_Admin, I want leads to be automatically distributed to my staff and to be able to reassign leads when needed so that no lead goes unattended.

#### Acceptance Criteria

1. WHEN a Lead is assigned to a Clinic (at creation or by Organization_Admin), THE Platform SHALL automatically assign the Lead to an active Staff member with lead access in that Clinic using Round_Robin rotation.
2. THE Round_Robin rotation SHALL cycle through all active Staff members with lead access in the Clinic in a consistent order, distributing leads evenly.
3. WHEN an Organization_Admin assigns a Clinic to an Unassigned_Lead, THE Platform SHALL trigger Round_Robin distribution within that Clinic.
4. THE Platform SHALL allow a Clinic_Admin to manually reassign a Lead to another active Staff member within the same Clinic, provided the Lead's status is `NEW` or `LOST`.
5. THE Platform SHALL NOT allow reassignment of Leads in `CONTACTED`, `QUALIFIED`, `CONVERTED`, or `LOST` status by Clinic-level Staff (only Clinic_Admin can reassign `LOST`).
6. WHEN a Lead is reassigned, THE Platform SHALL record the change in the Audit_Log including the previous and new assignee.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Reassigning a Lead not in `NEW` or `LOST` status | `INVALID_LEAD_REASSIGNMENT` |
| Recipient Staff member is inactive | `RECIPIENT_INACTIVE` |
| Recipient Staff member belongs to a different Clinic | `FORBIDDEN` |
| No active Staff with lead access in Clinic | `NO_ELIGIBLE_STAFF` |

#### Correctness Properties

- For any Lead L assigned to Clinic C, `L.assignedStaffId` SHALL reference an active Staff member in Clinic C with lead access.
- Round_Robin distribution SHALL result in an even distribution of leads across eligible Staff members over time.
- After reassignment, `L.assignedStaffId` SHALL reference the new assignee and the Audit_Log SHALL record the previous assignee.

### LM-6: Unassigned Lead Management

**User Story:** As an Organization_Admin, I want to view and assign unassigned leads to clinics so that no lead is lost due to missing clinic assignment.

#### Acceptance Criteria

1. THE Platform SHALL allow Organization_Admins to view all Unassigned_Leads across the Organization.
2. THE Platform SHALL allow Organization_Admins to assign a Clinic to any Unassigned_Lead.
3. WHEN a Clinic is assigned to an Unassigned_Lead, THE Platform SHALL trigger Round_Robin distribution within that Clinic.
4. Clinic-level Staff SHALL NOT see Unassigned_Leads.
5. WHEN an Unassigned_Lead is assigned a Clinic, THE Platform SHALL record the change in the Audit_Log.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Clinic-level Staff attempting to view Unassigned_Leads | `FORBIDDEN` |
| Assigning a Clinic from a different Organization | `FORBIDDEN` |

#### Correctness Properties

- For any Unassigned_Lead L, L SHALL NOT appear in any Clinic-level Staff member's lead list.
- After a Clinic is assigned to L, `L.clinicId` SHALL be non-null and `L.assignedStaffId` SHALL be set via Round_Robin.

### LM-7: Lead Actions and Notes

**User Story:** As a Staff member with lead edit permission, I want to log interactions against a lead so that the team has a complete history of all communications.

#### Acceptance Criteria

1. THE Platform SHALL allow Staff to add Lead_Actions to a Lead record, each with: `actionType` (from dropdown), optional `statusChange`, and `content` text.
2. THE Platform SHALL support the following `actionType` values: `WHATSAPP`, `EMAIL`, `FACEBOOK_MESSAGE`, `PHONE_CALL`, `IN_PERSON_MEETING`, `OTHER`.
3. WHEN a Lead_Action is added, THE Platform SHALL record: `actionType`, `content`, `timestamp` (UTC), and the Staff member who added it.
4. THE Platform SHALL display all Lead_Actions for a Lead in chronological order.
5. THE Platform SHALL preserve all Lead_Actions until the Lead is converted to a Patient.
6. WHEN a Lead is converted to a Patient, THE Platform SHALL transfer all Lead_Actions to the Patient record for historical reference.
7. WHEN a Lead_Action is added, THE Platform SHALL record the action in the Audit_Log.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Invalid `actionType` value | `VALIDATION_ERROR` (field: `actionType`) |
| Lead not found | `LEAD_NOT_FOUND` |

#### Correctness Properties

- For any two Lead_Actions A1 and A2 on the same Lead where `A1.timestamp < A2.timestamp`, A1 SHALL appear before A2 in the list.
- For any Lead L with N Lead_Actions, after conversion to Patient P, P SHALL have at least N historical records corresponding to L's Lead_Actions.

### LM-8: Lead Status and Priority Management

**User Story:** As a Staff member with lead edit permission, I want to update a lead's status and priority so that the team always knows where each prospect is in the pipeline.

#### Acceptance Criteria

1. THE Platform SHALL support `Lead_Status` values: `NEW`, `CONTACTED`, `QUALIFIED`, `CONVERTED`, `LOST`.
2. THE Platform SHALL allow Staff to change a Lead's status to any value except `CONVERTED` (set only by the conversion process).
3. WHEN a Lead's status is changed to `LOST`, THE Platform SHALL require explicit confirmation (`confirmed: true` in the mutation input).
4. THE Platform SHALL allow Staff to set and update `priority` and `tags`.
5. WHEN status, priority, or tags are changed, THE Platform SHALL record the change in the Audit_Log.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Attempting to set status to `CONVERTED` manually | `INVALID_STATUS_TRANSITION` |
| Changing status of already-CONVERTED Lead | `LEAD_ALREADY_CONVERTED` |
| Setting status to `LOST` without confirmation | `CONFIRMATION_REQUIRED` |
| Lead not found | `LEAD_NOT_FOUND` |

#### Correctness Properties

- After a Lead L is converted to a Patient: `L.status = CONVERTED` and SHALL NOT be changeable by any Staff action.
- `CONVERTED` status SHALL only be set via the conversion process in LM-9.

### LM-9: Lead Conversion to Patient

**User Story:** As a Staff member with lead convert permission, I want to convert a qualified lead to a patient so that the clinic can begin formal treatment without re-entering their information.

#### Acceptance Criteria

1. THE Platform SHALL allow Staff to convert a Lead to a Patient from the Lead profile page via a `convertLeadToPatient` mutation.
2. WHEN a Lead is converted, THE Platform SHALL auto-populate the Patient profile fields from the Lead's data.
3. WHEN a Lead is converted, THE Platform SHALL set the Lead's status to `CONVERTED`.
4. IF the Lead's `email` already exists as a Patient's `email` in the same Clinic, THE Platform SHALL block the conversion.
5. IF the Lead's `phone` already exists as a Patient's `phone` in the same Clinic, THE Platform SHALL block the conversion.
6. WHEN a Lead is successfully converted, THE Platform SHALL emit a `LeadConverted` event and record the conversion in the Audit_Log including the resulting Patient ID.
7. WHEN a Lead is converted, THE Platform SHALL transfer all Lead_Actions and the `Selfie_Analysis_Report` (if present) to the Patient record.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Lead's email matches existing Patient email | `CONVERSION_DUPLICATE_EMAIL` |
| Lead's phone matches existing Patient phone | `CONVERSION_DUPLICATE_PHONE` |
| Lead already converted | `LEAD_ALREADY_CONVERTED` |
| Lead not found | `LEAD_NOT_FOUND` |

#### Correctness Properties

- For any Lead L converted to Patient P: every field present in L that maps to a Patient field SHALL be present and equal in P immediately after conversion.
- For any Lead L where `L.email` matches an existing Patient's email in the same Clinic: conversion SHALL fail and no Patient record SHALL be created.
- After conversion: `L.status = CONVERTED` and SHALL NOT be changed by any Staff action.

### LM-10: Lead Appointment Booking

**User Story:** As a Staff member with appointment create permission, I want to book an appointment directly from a lead's profile so that I can schedule consultations without switching modules.

#### Acceptance Criteria

1. THE Platform SHALL allow Staff to initiate an appointment booking from the Lead profile page.
2. WHEN booking from a Lead profile, THE Platform SHALL pre-populate the appointment form with the Lead's name and contact details.
3. WHEN an appointment is booked for a Lead, THE Platform SHALL send an email notification to the Lead's email address.
4. THE Platform SHALL allow a Lead to book their own appointment via the Selfie_Analysis web component after receiving their `Selfie_Analysis_Report`.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Lead not found | `LEAD_NOT_FOUND` |
| No available slots | `SLOT_NOT_AVAILABLE` |

#### Correctness Properties

- For any appointment booked from Lead L, the appointment's contact fields SHALL equal L's corresponding fields at the time of booking.

### LM-11: Web Component and Webhook Configuration Access

**User Story:** As an Organization_Admin, I want exclusive control over web component setup and webhook configuration so that integration points are managed at the organization level.

#### Acceptance Criteria

1. THE Platform SHALL restrict the following actions to Organization_Admins only:
   - Creating, editing, and deleting Webhook_Sources and their Field_Mappings
   - Generating and revoking the Organization API key used by the Selfie Analysis and Appointment Booking web components
   - Configuring the Selfie Analysis web component (embed settings, clinic list display)
   - Configuring the Appointment Booking web component (embed settings)
2. Clinic_Admins and other Clinic-level Staff SHALL NOT have access to any of the above configuration actions.
3. WHEN any of the above configurations are changed, THE Platform SHALL record the change in the Audit_Log.

#### Failure Cases

| Condition | Error Code |
|-----------|------------|
| Clinic-level Staff attempting any configuration action | `FORBIDDEN` |

#### Correctness Properties

- For any configuration mutation on Webhook_Sources, API keys, or web component settings, the actor SHALL be an Organization_Admin or the mutation SHALL fail.
